# lyra-docs

## What it does

Reads a codebase and writes documentation to `docs/` at the project root. The documentation holds only knowledge the code itself cannot express: how the pieces fit together, each module's responsibility (derived from its public surface and callers, not inferred intent), and decisions the code makes explicit. 
It does not re-describe what every function does — that belongs next to the code and rots the moment it changes.

The skill is read-only against the source. Every prose claim cites a file and line range. If a claim cannot be traced to specific lines, it is omitted. 
The skill documents observable behavior and explicit decisions, never inferred purpose — a citation does not make an inference true.

Output is plain markdown, no emojis, no marketing language, no boilerplate openers. Each file passes a council review gate before it lands.

## The docs/ tree

```
docs/
├── README.md              ← general index, entry point
├── architecture.md        ← how the modules wire together
├── directory-tree.md      ← annotated source tree
├── modules/
│   └── {module}.md        ← one file per logical module
├── api/                   ← only if a public API exists
│   └── {endpoint}.md
└── decisions.md           ← only if the code makes choices explicit (see below)
```

Folders are generated only for code that exists. No `api/` folder if there is no public surface. No `decisions.md` if the code reveals no explicit decisions. Do not create empty folders.

## What does not belong in docs/

README material (install, quickstart), per-function docstrings, changelogs, licenses, and any prose that paraphrases what a file's symbols already declare. If a paragraph could be regenerated by reading the code, cut it.

## What counts as an explicit decision

`decisions.md` records only choices the code states directly: comments like `// We chose X over Y because Z`, ADR files, config files with explanatory comments, or commit messages that document a trade-off. A non-obvious pattern you notice is an observation, not a decision — it goes in `architecture.md` with a citation, not in `decisions.md`.

## Scope and scale

For monorepos, generate one `docs/` at the root with `modules/` files per workspace package. For multiple languages in one repo, derive module boundaries from the directory structure each language uses (packages, namespaces, import paths). Skip generated code committed to the repo — detect it by `// generated` headers, `gen/` or `generated/` paths, or build-tool config. 
Document config files only when they encode non-obvious behavior. Document tests only when they reveal intent the source does not.

For repos over 500 source files, sample strategically: read all entry points, all public interfaces, and representative files from each module. 
Note the sampling in `docs/README.md` so the reader knows coverage is not exhaustive.

## The workflow

1. Glob the repo. Build the real file inventory. Skip `node_modules`, `.git`, build artifacts, lockfiles, and generated code.
2. Read every source file (or sample per the scale rule above). Full pass, no skipping within the chosen scope.
3. Derive module boundaries from the code's own structure (directories, packages, import paths), not from assumptions.
4. Write `directory-tree.md` from the glob result.
5. Draft each `modules/*.md` in dependency order, foundations first.
6. Run the per-file council gate on each draft (see below).
7. Stamp every file with the commit hash it was verified against. A line at the top: `Verified at commit <hash>`.
8. Write `architecture.md` once the modules are stable.
9. Write `docs/README.md` last. It indexes everything else, so it must reflect final state.

## The anti-hallucination protocol

Every prose claim gets a trailing citation: `file/path.ext:42` for a single line, `file/path.ext:42-58` for a range. No citation, no claim. If you cannot point to a line, omit the claim.

Never infer behavior the code does not show. Re-exported or dead code is labeled as such, not narrated as live. The verifier step re-opens each cited file and line and confirms the claim matches — mismatches force a rewrite.

For synthesis claims in `architecture.md` that span many files, cite the primary file and list the others in parentheses: `(also src/a.ts:10, src/b.ts:33)`.

## The anti-slop checklist

Run this before a file is done:

- Every claim cited to file and line. Citations spot-verified.
- No emojis.
- No banned vocabulary: leverage, utilize, seamless, robust, showcase, foster, delve, additionally, moreover, crucial, pivotal, enhance, unlock, elevate, harness, intricate, vibrant, testament, underscore, paradigm, synergy, ecosystem, innovative, cutting-edge.
- No repeated sentences or restated bullets across files.
- No invented names, types, or config keys.
- No boilerplate openers: "This document describes...", "It is worth noting...", "In today's...".
- No "This function is responsible for..." auto-generated phrasing.
- Active voice, plain verbs, no marketing.
- No placeholder `TODO` or `...` left in output.
- No hedge words: essentially, basically, it should be noted.
- Sentence-case headings, not Title Case.

## The per-file council gate

Each file goes through four steps before it lands. The review is run by `lyra-council` with a substituted advisor set — the five axes below replace the default personas.

1. Draft. Write the file.
2. Five-advisor review via `lyra-council`. The five advisors are:
   - Accuracy — does the prose match what the code does?
   - Citations — does every claim have a file:line, and does the line actually say that?
   - Slop and voice — any banned vocabulary, boilerplate openers, emojis, hedges?
   - Repetition — is anything restated from another file without earning its place?
   - Completeness — is anything irreducible missing from this file's scope?
3. Rewrite. Apply every blocking finding. Dispute non-blocking ones with reasons.
4. Verify. Re-check citations against source. The file is done only when zero blocking findings remain. Otherwise loop back to step 1.

The council session for each file is saved to `.lyra/council/sessions/` by lyra-council, and a review entry is appended to the verification log (see Memory below).

## Memory

lyra-docs persists verification state in `docs/.lyra/verification-log.json`. This is the memory that tracks what was verified, when, and against what commit. It lives in the repo (commit it to git) so the whole team shares the same verification state.

The log records, per document: the commit it was verified against, the exact citations (file + line range + content hash of the cited lines), when it was last checked, and the council review verdicts. It stores metadata only — paths, git hashes, timestamps, statuses. No file contents, no user data, no secrets.

### Drift detection

Run `lyra-docs drift` to check which docs are stale. The check compares each citation's stored content hash against the current source. Status levels, worst signal wins:

- `broken` — a cited file no longer exists
- `stale` — the cited lines changed
- `line_shift_suspect` — the file changed but the cited bytes are the same; lines may have shifted
- `modified` — the doc itself was hand-edited since verification
- `verified` — no drift detected

The report groups docs by status and shows the specific citations that triggered each. Exit code is non-zero if any doc is `broken` or `stale`, so CI can gate on it.

### Council reviews

Every council pass — whether the built-in five-advisor gate or an external `lyra-council` session — appends a review entry to the log: reviewer, verdict, consensus score, per-axis scores, and a reference to the full transcript in `docs/.lyra/council-log.json`. Drift status and review status are orthogonal: a doc can be `verified` (no drift) but `needs-revision` (council found issues), or `stale` (code moved) but `approved` (last known good).

## Composes with

- `lyra-council` — runs the per-file council gate. lyra-docs substitutes its five review axes (accuracy, citations, slop, repetition, completeness) for the default personas.
- `lyra-analyze-codebase` — run first to map the codebase. lyra-docs inherits its read-only discipline and the Grep-to-Read-to-cite loop.
- `lyra-clean-architecture` — when documenting `architecture.md`, invoke it to name the pattern (none, layered, hexagonal, DDD) rather than inventing one.
- `lyra-api-design` — when the `api/` folder exists, invoke it to describe contracts accurately.
- `humanizer` — its Wikipedia-based "Signs of AI writing" patterns inspired the anti-slop checklist. The checklist adapts those patterns to code documentation.
